Pengenalan

Apa itu Cakra dan untuk siapa?

Cakra adalah platform analisis drive test berbasis web yang berjalan sepenuhnya di browser — tanpa instalasi, tanpa backend server. Cakra mengolah file log dari berbagai tool drive test standar industri dan menyajikan hasil analisis berupa grafik interaktif, peta sinyal, distribusi KPI, dan deteksi titik rawan secara instan.

Multi-Tool Support
G-NetTrack Pro, TEMS, NEMO, SIGMON didukung dalam satu platform.
100% Lokal
Data tidak pernah meninggalkan browser Anda. Privasi terjamin.
Tanpa Instalasi
Buka browser, upload file, analisis selesai. Semudah itu.
✦
AI Analyst
Analisis otomatis dan chat berbasis Claude AI untuk insight mendalam.
Peta Interaktif
Visualisasi spasial RSRP/RSRQ/SNR dengan overlay events handover.
PWA Offline
Install sebagai aplikasi, gunakan tanpa koneksi internet.

Cara Mulai

Dari nol ke dashboard dalam 3 langkah

1
Siapkan File Drive Test
Export file log dari tool drive test Anda. G-NetTrack Pro: menu Session → Export TXT. TEMS/NEMO: export ke format CSV atau TXT standar. File KML events bersifat opsional namun direkomendasikan untuk analisis handover.
2

Buka halaman utama Cakra. Drag & drop file log ke zona upload kiri, atau klik untuk pilih file. Untuk KML, gunakan zona upload kanan. Cakra mendukung multi-file sekaligus — upload beberapa rute sekaligus!

Referensi Standar

Threshold kategorisasi "good/fair/poor" yang dipakai di Cakra adalah konvensi umum operator industri (selaras dengan vendor optimization manual seperti Ericsson, Huawei, Nokia) — 3GPP sendiri tidak menetapkan label kualitatif untuk nilai RSRP/RSRQ, hanya rentang valid pengukurannya. Algoritma deteksi (Coverage Gap, Cell Churn, dst) adalah heuristik engineering — bukan algoritma yang distandarisasi.

Coverage Gap Detection

Deteksi dan pemetaan area blank spot secara otomatis

Dasar Teori

Dalam jaringan LTE, kekuatan sinyal referensi yang diterima UE (User Equipment) diukur sebagai RSRP (Reference Signal Received Power). RSRP mengukur daya rata-rata sumber daya elemen (resource element) yang membawa sinyal referensi sel dalam band LTE. Nilai RSRP sangat bergantung pada jarak UE dari antena, penghalang fisik (gedung, bukit), dan konfigurasi antena (tinggi, tilt, azimuth).

Ketika RSRP turun di bawah nilai minimum operasional, UE tidak lagi dapat mempertahankan koneksi yang stabil. Area ini disebut coverage gap atau blank spot. Dalam konteks drive test, coverage gap teridentifikasi sebagai segmen rute di mana RSRP secara konsisten berada di bawah threshold yang ditentukan.

Catatan Threshold RLF
Tidak ada nilai RSRP tunggal yang resmi sebagai ambang Radio Link Failure (RLF) di 3GPP. RLF di 3GPP TS 36.331 didefinisikan via mekanisme N310/N311: yaitu N310 indikasi out-of-sync berturut-turut dari fisik layer, dengan kriteria out-of-sync biasanya mengacu pada Qout ≈ −8 dB SINR (BLER ~10% pada PDCCH). RSRP rendah (−110 sampai −120 dBm) memang sangat berkorelasi dengan RLF dalam praktik karena umumnya disertai SINR yang turun, tapi RSRP bukan kriteria langsung. Threshold default Coverage Gap di Cakra (−100 dBm) mengikuti praktik operator KPI premium, bukan ambang RLF.

Cara Kerja Algoritma

1
Ambil data GPS valid
Cakra memfilter semua data point yang memiliki koordinat GPS valid (latitude dan longitude bukan null). Data point tanpa GPS dikecualikan dari analisis spasial.
2
Bandingkan RSRP tiap titik terhadap threshold
Setiap data point dibandingkan dengan threshold yang dipilih (default −100 dBm). Titik dengan RSRP ≤ threshold ditandai sebagai "buruk". Threshold dapat dikonfigurasi dari −90 hingga −110 dBm.
3
Segmentasi berurutan
Titik-titik "buruk" yang berurutan (consecutive) dikelompokkan menjadi satu segmen gap. Segmen baru dimulai ketika ada titik "baik" di antara dua cluster titik buruk. Segmen dengan panjang ≥ 2 titik dianggap signifikan.
4
Visualisasi polyline di peta
Setiap segmen gap digambar sebagai polyline merah tebal di peta Leaflet, dengan popup yang menampilkan rata-rata RSRP, nilai minimum, panjang segmen, dan nama sel yang serving.
5
Tabulasi dan statistik
Panel Coverage Gap menampilkan tabel semua segmen dengan statistik agregat: jumlah gap, total titik, panjang rata-rata, dan segmen terpanjang.

Cara Membaca Hasil

Perhatikan tiga kolom utama pada tabel Coverage Gap: Avg RSRP menunjukkan rata-rata kualitas di segmen tersebut, Min RSRP menunjukkan titik terburuk, dan Panjang menunjukkan berapa banyak data point yang masuk dalam segmen. Semakin panjang segmen dan semakin rendah Min RSRP, semakin prioritas segmen ini untuk ditangani.

Pertimbangan Penting
Coverage gap yang pendek (2–5 titik) di area dengan kecepatan tinggi bisa jadi hanya shadow fading sesaat, bukan blank spot struktural. Bandingkan dengan data historis atau lakukan re-drive test untuk konfirmasi. Gap yang konsisten muncul di lokasi yang sama di beberapa sesi drive test mengindikasikan masalah struktural coverage.

Cell Churn / Handover Instability

Identifikasi area dengan serving cell yang berputar cepat — penyebab bisa beragam

Catatan Penting: Bukan Pilot Pollution Klasik
Versi awal Cakra menamakan fitur ini "Pilot Pollution Detection". Setelah review terhadap definisi industri, nama itu kurang tepat. Pilot pollution klasik (per Ericsson Cell Planning Principles, Huawei RNO Manual, GSMA optimization guide) didefinisikan sebagai kondisi di mana ≥3 sel memiliki RSRP yang berada dalam window ΔdB (umumnya 3–6 dB) dari strongest cell, terukur bersamaan di satu titik, tanpa server dominan. Pengukuran ini butuh data neighbor cell measurement (multi-cell scanner data atau mode pengukuran neighbor di UE) — bukan sekadar mencatat serving cell yang berganti-ganti.

File log G-NetTrack Pro default tidak mengandung neighbor measurement. Yang Cakra deteksi adalah cell churn: area di mana serving cell sering berganti dalam waktu singkat. Ini bisa disebabkan oleh pilot pollution, tapi juga oleh penyebab lain yang harus dibedakan.

Apa yang Sebenarnya Dideteksi

Cakra menggunakan pendekatan Sliding Window Serving Cell Analysis: untuk setiap titik data ke-i, algoritma melihat 10 titik sebelumnya dan menghitung berapa banyak cellname unik yang menjadi serving cell dalam window itu. Jika ≥ 3 sel berbeda muncul, titik tersebut di-flag sebagai cell churn. Ini menangkap area instabilitas serving cell secara temporal.

Konteks: Definisi RSRQ untuk Pilot Pollution Klasik

Rumus RSRQ
RSRQ = N × (RSRP / RSSI)
Di mana N adalah jumlah resource block (RB) yang digunakan dalam pengukuran. Pada area dengan banyak sel yang bersaing kuat, RSSI (total wideband received power) meningkat sehingga RSRQ turun, meskipun RSRP serving cell tinggi. Karena itu RSRQ adalah indikator kunci pilot pollution sebenarnya — bukan jumlah serving cell yang berganti.

Untuk verifikasi: kalau Anda melihat cell churn di Cakra dan RSRQ di area itu di bawah −13 dB sambil RSRP cukup baik, itu indikator kuat pilot pollution. Kalau RSRQ-nya normal, kemungkinan besar penyebabnya bukan pilot pollution.

Throughput Correlation

Korelasi antara kualitas sinyal RF dan performa throughput pengguna

Dasar Teori

Dalam jaringan LTE, throughput downlink (DL) secara teoritis berkorelasi positif dengan kualitas sinyal — semakin baik RSRP dan SINR, semakin tinggi Modulation and Coding Scheme (MCS) yang bisa digunakan, sehingga throughput meningkat. Namun dalam praktik lapangan, korelasi ini tidak selalu sempurna karena banyak faktor non-RF yang juga mempengaruhi throughput.

Cakra menggunakan Koefisien Korelasi Pearson (r) untuk mengukur kekuatan hubungan linear antara RSRP dan DL throughput. Nilai r berkisar antara −1 (korelasi negatif sempurna) hingga +1 (korelasi positif sempurna). Nilai r = 0 berarti tidak ada hubungan linear.

Formula Korelasi Pearson
r = Σ[(xi − x̄)(yi − ȳ)] / √[Σ(xi − x̄)² × Σ(yi − ȳ)²]
Di mana x = RSRP setiap titik, y = DL throughput setiap titik, x̄ = rata-rata RSRP, ȳ = rata-rata throughput.

Cara Membaca Nilai Korelasi

Nilai rInterpretasiImplikasi Jaringan
r > 0.7Korelasi kuat positifThroughput sangat bergantung pada kondisi RF. Perbaikan coverage (RSRP) akan langsung meningkatkan user experience. Masalah throughput = masalah RF.
r = 0.4 – 0.7Korelasi sedangRF berkontribusi tapi ada faktor lain. Perlu investigasi lanjutan: apakah ada masalah congestion, transport, atau parameter scheduler?
r = 0.2 – 0.4Korelasi lemahThroughput rendah bukan terutama karena RF. Kemungkinan: congestion di backhaul, transport bandwidth terbatas, atau isu di core network.
r < 0.2Tidak berkorelasi / korelasi sangat lemahMasalah throughput hampir pasti bukan RF. Investigasi ke arah core/transport, atau kemungkinan test environment issue (speedtest server bottleneck).

Analisis per Bucket RSRP

Selain korelasi global, Cakra menampilkan rata-rata DL throughput yang dikelompokkan per kategori RSRP (bucket). Cara ini lebih intuitif bagi engineer karena langsung terlihat: "di area dengan RSRP > −80 dBm, rata-rata DL throughput adalah X Mbps." Jika grafik bucket ini menunjukkan pola kenaikan monoton (throughput naik seiring RSRP naik), korelasi RF-throughput memang kuat. Jika plateau atau acak, ada faktor lain yang dominan.

Faktor Non-RF yang Mempengaruhi Throughput

FaktorGejala di DataCara Identifikasi

Tip Analisis Lanjutan
Jika korelasi RF-throughput lemah (r < 0.4), gunakan fitur AI Analyst → Throughput untuk analisis lebih mendalam. AI akan mengevaluasi pola throughput vs waktu, mencari pola congestion, dan memberikan rekomendasi investigasi lanjutan.

Visualisasi di Peta: Layer Throughput

Layer Throughput di peta menampilkan setiap titik GPS dengan warna dan ukuran marker yang proporsional terhadap DL throughput di titik tersebut. Marker besar berwarna hijau = throughput tinggi, marker kecil berwarna merah = throughput rendah. Overlay layer ini di atas layer RSRP (dengan toggle) memungkinkan inginer untuk langsung melihat apakah area dengan RSRP rendah berkorespondensi dengan area throughput rendah.

Site-based View

Agregasi KPI per eNodeB untuk identifikasi site bermasalah

Konsep dan Motivasi

Analisis drive test konvensional menyajikan data secara kronologis (time-series) atau spasial (per GPS point). Site-based View mengubah perspektif ini: data dikelompokkan per eNodeB (eNB) atau per site, sehingga engineer dapat langsung melihat performa agregat dari setiap site yang dilalui rute drive test.

Ini sangat berguna karena dalam satu sesi drive test, sebuah site bisa melayani UE di beberapa titik berbeda sepanjang rute. Dengan agregasi per site, bisa terlihat site mana yang secara konsisten memberikan RSRP rendah, DL throughput rendah, atau banyak titik rawan — hal yang tidak mudah dilihat dari tabel time-series biasa.

Cara Kerja Agregasi

Cakra mengidentifikasi site dari dua sumber data secara prioritas:

1
Dari field eNodeB/Node ID (prioritas utama)
G-NetTrack Pro menyimpan ID eNodeB di kolom node. Jika tersedia, ini adalah pengelompokan paling akurat karena langsung mengidentifikasi base station yang serving, terlepas dari nama sel.
2
Dari prefix nama sel (fallback)
Jika node ID tidak tersedia, Cakra mengekstrak prefix dari nama sel (cell name) dengan menghapus suffix numerik (misal: SITE_ABC_1, SITE_ABC_2, SITE_ABC_3 → semua dikelompokkan ke SITE_ABC). Ini mengasumsikan konvensi penamaan sel yang konsisten, yang umumnya berlaku di operator Indonesia.
3
Hitung KPI per site
Untuk setiap grup site, dihitung: rata-rata RSRP (dari semua data point yang dilayani site tersebut), rata-rata DL throughput, jumlah sektor/cell yang teridentifikasi, jumlah titik rawan (RSRP < −100 atau RSRQ < −19 atau SNR < −10), dan total data point.

Prioritisasi Site untuk Tindak Lanjut

Urutkan tabel berdasarkan persentase titik rawan dari tinggi ke rendah. Site dengan kombinasi Avg RSRP rendah + persentase titik rawan tinggi + data point banyak adalah kandidat prioritas untuk investigasi teknis dan field visit.

Validasi Silang dengan OMC
Setelah mengidentifikasi site bermasalah dari Site-based View, lakukan validasi silang dengan data OMC (Operations and Maintenance Center). Cek apakah ada alarm aktif, degradasi PDCP throughput, atau anomali di counter KPI seperti RRC Setup Success Rate atau E-RAB Drop Rate. Kombinasi drive test data dan OMC counter memberikan gambaran paling komprehensif.

Keterbatasan dan Catatan

Site-based View memiliki beberapa keterbatasan yang perlu dipahami. Pertama, jumlah data point per site sangat bergantung pada rute drive test — site yang dilewati singkat memiliki sedikit data point, kurang representatif. Kedua, dalam area handover yang aktif, UE mungkin dilayani bergantian oleh dua sel dari site berbeda, sehingga performa bisa ter-split. Ketiga, untuk site dengan coverage sangat luas (seperti di area rural), mungkin tidak semua area coverage-nya tercakup oleh rute drive test.

Penggunaan Terbaik
Site-based View paling efektif digunakan untuk pre-/post-optimization comparison — lakukan drive test sebelum dan sesudah perubahan parameter/hardware di site tertentu, lalu bandingkan Avg RSRP dan persentase titik rawan dari site tersebut di kedua sesi. Kenaikan Avg RSRP > 3 dB atau penurunan titik rawan > 20% adalah indikasi optimasi berhasil.

Standar KPI

Threshold konvensi operator industri (selaras dengan rentang valid 3GPP)

Asal Threshold
Tabel di bawah adalah konvensi umum yang dipakai operator dan vendor optimization (Ericsson, Huawei, Nokia, ZTE) — bukan standar formal 3GPP. 3GPP TS 36.133/38.215 mendefinisikan rentang valid pengukuran (RSRP −140…−44 dBm, RSRQ −19,5…−3 dB Rel-14, dst) tapi tidak memberi label "Sangat Baik/Buruk". Threshold di Cakra di-tune untuk drive test di area urban Indonesia. Sesuaikan kalau use case Anda berbeda (rural, indoor, IoT/NB-IoT, dll).

FAQ

Pertanyaan yang sering diajukan

Apakah data saya aman?

Ya. Seluruh pemrosesan dilakukan 100% di browser Anda. Data drive test tidak pernah dikirim ke server manapun (kecuali ke Anthropic API jika menggunakan fitur AI Analyst, dengan izin eksplisit Anda).

Berapa batas ukuran file?

Tidak ada batas ukuran file yang keras. Namun untuk performa optimal, file di bawah 100MB direkomendasikan. Untuk file sangat besar (>50k baris), data otomatis di-thinning untuk rendering.

Mengapa peta tidak muncul?

Peta memerlukan koneksi internet untuk memuat tile basemap dari CARTO. Pastikan perangkat terhubung ke internet saat pertama kali membuka dashboard.

Format file apa saja yang didukung untuk KML?

Cakra mendukung file KML standar dari G-NetTrack Pro dengan events type HANDOVER_DATA_4G4G dan CELL_RESELECTION_4G4G. KML dari tool lain mungkin tidak dikenali.

Bisakah digunakan untuk data 5G NR?

Ya. Cakra mendukung parameter 5G NR termasuk NR-RSRP dan SS-SINR, untuk file yang mengandung data 5G NR dari tool yang didukung.

Masih ada pertanyaan?
Gunakan fitur AI Analyst di dashboard untuk konsultasi teknis langsung. AI dapat menjawab pertanyaan spesifik tentang data drive test Anda.
